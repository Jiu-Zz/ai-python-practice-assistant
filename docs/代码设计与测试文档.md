# AI 驱动的 Python 编程练习助手代码设计与测试文档

## 1. 文档目的

本文档用于说明当前项目代码的实现设计、模块划分、数据存储方式、核心流程和测试方案，作为项目验收、后续维护和再次开发的依据。

项目采用前后端分离结构：

- 前端：React + Vite + TypeScript
- 后端：Python FastAPI + SQLAlchemy
- 数据库：SQLite，本地默认路径为 `code/backend/data/app.db`
- 代码执行：后端受限 Python 子进程
- AI 助手：后端统一封装大模型调用，根据题目、代码、评测结果生成结构化诊断与学习提示

## 3. 总体架构设计

系统分为浏览器前端、FastAPI 后端和 SQLite 数据库三部分。

前端负责页面展示、用户交互、代码输入、登录态保存和 API 调用。后端负责认证、题目管理、代码运行、提交评测、AI 诊断记录、学习推荐和数据持久化。数据库负责保存用户、题目、测试用例、运行记录、提交记录、AI 记录和学习推荐等数据。

核心调用链如下：

```text
用户浏览器
  -> React 前端页面
  -> src/api/client.ts 统一请求封装
  -> FastAPI /api/v1 接口
  -> router 路由层
  -> service 业务层
  -> repository 数据访问层
  -> SQLite 数据库
```

代码运行和提交评测流程会额外调用 `CodeRunner`，由后端启动受限 Python 子进程执行用户代码，并将 stdout、stderr、错误类型和耗时返回给业务层。

## 4. 后端代码设计

### 4.1 应用入口

后端入口文件为 `code/backend/app/main.py`。

主要职责：

- 创建 FastAPI 应用
- 配置 CORS，允许前端 `5173` 端口访问
- 启动时初始化数据库表
- 启动时写入演示种子数据
- 注册认证、用户、题目、AI、推荐相关路由
- 注册统一异常处理
- 提供 `/health` 健康检查接口

后端 API 前缀统一为 `/api/v1`。

### 4.2 配置模块

主要配置项：

| `DATABASE_URL` | 数据库连接地址 | `sqlite:///./data/app.db` |

### 4.7 代码运行器设计

代码运行器文件为 `code/backend/app/infrastructure/code_runner.py`。

`CodeRunner` 的执行过程：

1. 使用 AST 解析用户代码。
2. 拦截不安全模块和函数，例如 `os`、`subprocess`、`socket`、`open`、`eval`、`exec`。
3. 将用户代码写入临时目录下的 `main.py`。
4. 使用当前 Python 解释器以隔离模式 `-I` 执行。
5. 捕获 stdout、stderr、返回码和超时。
6. 根据结果返回 `success`、`runtime_error` 或 `time_limit`。
7. 从 stderr 中提取 `SyntaxError`、`TypeError`、`ZeroDivisionError` 等错误类型。

当前实现适合课程演示和基础题目练习，不等同于生产级沙箱。若后续面向真实多用户环境，应替换为 Docker 沙箱或远程评测服务。

### 4.8 提交评测设计

提交评测由 `JudgeService.submit_code()` 实现。

流程如下：

1. 根据 `problem_id` 查询已发布题目。
2. 读取该题目的全部测试用例。
3. 创建一条 `Submission` 提交记录。
4. 逐个测试用例调用 `CodeRunner.run()`。
5. 比较实际输出和期望输出。
6. 保存每个测试用例的 `TestResult`。
7. 计算通过数量、总数量、分数、耗时和最终状态。
8. 返回提交结果给前端。

评测状态规则：

| 状态 | 说明 |
| --- | --- |
| `accepted` | 全部测试用例通过 |
| `wrong_answer` | 程序正常运行，但输出不符合期望 |
| `runtime_error` | 执行出现异常 |
| `time_limit` | 执行超时 |

### 4.9 AI 助手设计

AI 相关代码位于：

- `code/backend/app/services/ai_tutor_service.py`
- `code/backend/app/schemas/ai.py`
- `code/frontend/src/components/AiPanel.tsx`

设计实现方式：

1. 前端将当前题目 ID、用户代码、错误信息、失败用例和提交 ID 发送到后端。
2. 后端 `AiTutorService` 查询题目信息。
3. 后端整理题目描述、输入输出要求、用户代码、运行错误、失败用例等上下文。
4. 后端通过统一的 LLM 客户端构造 Prompt，请求千问大模型生成诊断内容。
5. 大模型返回结构化结果，内容包含错误类型、问题总结、可能原因、排查步骤、相关知识点和必要的参考代码片段。
6. 后端对模型响应进行结构化校验和长度控制，异常情况下使用本地兜底策略保证接口可用。
7. 后端将 AI 响应保存到 `ai_records` 表，便于后续查看和学习记录分析。
8. 前端 AI 面板展示讲解内容、分步分析和代码块。

以“判断奇偶”题目为例，AI 诊断会围绕题目要求生成如下讲解内容：

```text
需求理解：接收一个整数，判断奇偶。能被 2 整除是偶数 (even)，不能则是奇数 (odd)。
核心判断逻辑：整数对 2 取余，n % 2 == 0 -> 偶数，n % 2 != 0 -> 奇数
代码分步思路：读取键盘输入，转为整数 n = int(input())
第一步：读取键盘输入，转为整数 n = int(input())
第二步：用 if 条件判断取余结果
第三步：按规则输出对应字符串
```

对应参考代码为：

```python
n = int(input())
if n % 2 == 0:
    print("even")
else:
    print("odd")
```
